# Portfolio Content Guide

This guide defines the exact content strategy Codex should preserve during implementation.

## Core positioning

Use this positioning across the portfolio:

```txt
Software Engineer building full-stack, real-time, and operational software for complex industries.
```

Alternative short hero wording:

```txt
I build systems that solve real problems at scale.
```

Supporting line:

```txt
I build full-stack software across frontend, backend, real-time workflows, and cloud-connected platforms for industries where reliability matters.
```

## Voice

The voice should be:

- precise;
- confident;
- technical;
- not inflated;
- not junior;
- not corporate nonsense;
- not fake-humble.

The writing should make Siddeg sound like a serious engineer who has worked inside real operational systems.

## Claims policy

Do not invent.

Use facts from the CV and supplied notes only.

Allowed facts:

- 3+ years experience;
- worked across IoT, fintech, healthcare, industrial automation, fuel operations, and public web platforms;
- React, Next.js, TypeScript, Node.js, FastAPI, Django, PostgreSQL, Redis, AWS, Docker, WebSockets, ThingsBoard;
- Saudi-market and Arabic/RTL product context;
- Inseejam platform live across 20+ SPL branches;
- Real-Time Cash Management System targeted reconciliation reduction from 2+ hours to under 5 minutes;
- offline-first cash system designed for 10,000+ local transactions;
- petroleum platform included 55+ API endpoints according to CV;
- no personal photo;
- confidential projects must not expose client screenshots or proprietary details.

Avoid unsupported claims:

- exact number of users unless supplied;
- exact revenue generated unless supplied;
- invented Lighthouse scores;
- invented conversion improvements;
- claims that Siddeg built entire platforms alone when work was collaborative;
- fake company names or hidden client names.

## Confidential-project disclaimer

Use this on confidential project pages:

```txt
This case study avoids client-sensitive screens, internal data, and proprietary implementation details. Visuals are representative system diagrams created to explain the workflow safely.
```

## Public projects

Public projects can include:

- live link;
- screenshots;
- direct frontend implementation description;
- responsive design notes;
- public proof positioning.

Public project wording should show frontend quality without pretending they are deeper than they are.

## Confidential projects

Confidential projects should include:

- system purpose;
- problem;
- Siddeg’s role;
- frontend and backend contribution areas;
- technical environment;
- representative workflow or architecture diagram;
- impact from supplied material only.

Confidential project visuals should be diagrams, not fake screenshots.

## Words to avoid

Avoid overused portfolio words unless necessary:

- passionate
- pixel-perfect
- beautiful experiences
- innovative solutions
- cutting-edge
- seamless
- dynamic
- highly motivated
- creative developer

Use stronger specific words:

- operational
- reconciliation
- telemetry
- role-based
- offline-first
- event-driven
- auditability
- reliability
- workflow
- system boundary
- data integrity
- frontend/backend modules

## Final content check

Before shipping, every line should pass this test:

```txt
Could Siddeg defend this in an interview?
```

If not, rewrite it.
